## P0-5-billing-rls — Lock billing/credits RLS: drop WITH CHECK(true) policies + revoke client writes on subscriptions, subscription_usage, voice_credits

**Verify verdict:** yes  

**Required changes:** None required. Recommended (already noted in the proposal, optional): after deploy run SELECT polname, cmd, qual, with_check FROM pg_policies WHERE tablename IN ('subscriptions','subscription_usage','voice_credits') to confirm no extra permissive policy survived schema drift, and audit existing rows for prior tampering (plan_type<>'trial' with NULL stripe_subscription_id; voice_credits source='purchase' with NULL stripe_payment_id).


**Breaks?:** No legitimate flow is broken. The one real risk — the client-side onboarding UPDATE (useSubscription.ts:155, Dashboard.tsx:478) — is explicitly preserved via the column-level GRANT UPDATE (onboarding_completed) plus an own-row UPDATE policy. Service-role billing writes, client reads, developer dashboards, and SECURITY DEFINER limit functions are all unaffected. Only residual: if prod has drifted policy names, DROP IF EXISTS by repo names may leave an unknown permissive policy (proposal flags this and recommends a pg_policies post-deploy check).


**Rationale:** The original migrations created INSERT `WITH CHECK(true)` and UPDATE `USING(true)` policies with no `TO` clause (so they apply to PUBLIC, including `authenticated`) on `subscriptions` (20251215071210_*.sql:47-53), `subscription_usage` (:60-66), and `voice_credits` (20251216190601_*.sql:26-35). I confirmed across all 73 migrations these were never dropped. Because the client ships only the publishable key, RLS is the sole boundary, so any logged-in user can `UPDATE subscriptions SET plan_type='enterprise', status='active'` or `INSERT voice_credits` with arbitrary minutes — a complete Stripe bypass — and `USING(true)` has no `user_id` scope so one tenant can overwrite/void another tenant's billing.

All legitimate billing writes go through service-role edge functions (verified: sync-subscription, stripe-webhook, deduct-voice-minutes, purchase-voice-credits, verify-voice-credits-purchase, get-subscription all build a `supabaseAdmin` client from `SUPABASE_SERVICE_ROLE_KEY` and do every `.insert/.update/.upsert` through it). Service-role bypasses RLS AND table grants, so dropping these policies and revoking client writes costs those flows nothing.

IMPORTANT correction to the audit's "zero functional cost" claim: I found TWO live client-side UPDATEs to `subscriptions` from the browser — `src/hooks/useSubscription.ts:156` (`completeOnboarding`: `.update({ onboarding_completed: true }).eq('user_id', user?.id)`) and `src/pages/Dashboard.tsx:479` (dev-only `handleTestOnboarding`: `.update({ onboarding_completed: false }).eq('user_id', user.id)`). Both only touch the harmless `onboarding_completed` flag on the caller's own row. A blanket REVOKE UPDATE on `subscriptions` would break the onboarding wizard. So instead of a blanket revoke on `subscriptions`, the fix uses the Postgres-native column-privilege mechanism that RLS itself cannot provide: it REVOKEs UPDATE on ALL columns from authenticated, then GRANTs UPDATE on ONLY `onboarding_completed`, and replaces the `USING(true)` policy with an own-row `USING(auth.uid()=user_id) WITH CHECK(auth.uid()=user_id)` policy. Result: the client can still flip its own onboarding flag but cannot write `plan_type`, `status`, `stripe_*`, `amount`, etc. The BEFORE-UPDATE `update_updated_at_column` trigger that sets `NEW.updated_at` is unaffected, because Postgres checks column UPDATE privilege against the statement's SET-list columns, not columns mutated by a trigger.

`subscription_usage` and `voice_credits` have NO client writes anywhere in src/ (verified by grep — only SELECTs), so they get the full treatment: drop permissive INSERT/UPDATE policies and REVOKE INSERT/UPDATE/DELETE entirely from anon and authenticated.

All existing SELECT-own-row policies (auth.uid()=user_id) and the developer SELECT policies (20260104033710_*.sql) are SELECT-only and are left intact, so the client can still read its plan/usage/credits and the SECURITY DEFINER limit functions (subscription_plan_for_limits, etc.) continue to work (they run as definer and read subscriptions independent of caller grants).


**Risks:** Behavioral changes for the client: (1) any future client code that tries to INSERT/DELETE on these tables, or UPDATE any column of subscriptions other than onboarding_completed, will now get a permission error (42501) — this is the intended hardening; if some unseen client path relies on it, it will surface as a denied write rather than silent corruption. (2) Column-level GRANT/REVOKE granularity: `REVOKE UPDATE ON public.subscriptions FROM authenticated` followed by `GRANT UPDATE (onboarding_completed)` is the correct sequence; if a future migration runs a bare `GRANT UPDATE ON public.subscriptions TO authenticated` it would silently re-widen access. (3) If a stale prod policy with a different name exists (schema drift between repo and prod, per MEMORY), the DROP POLICY IF EXISTS by the repo names may leave an unknown permissive policy in place — recommend running `SELECT polname, cmd, qual, with_check FROM pg_policies WHERE tablename IN ('subscriptions','subscription_usage','voice_credits')` post-deploy to confirm only SELECT/own-row-UPDATE policies remain. (4) This migration assumes RLS is already enabled (it is, from the base migrations); it does not re-assert ENABLE ROW LEVEL SECURITY. All statements are idempotent-safe (DROP ... IF EXISTS, REVOKE is a no-op if already revoked) except the CREATE POLICY, which will error if re-run — standard for forward-only migrations.

## P0-6-candidate-cols — Block candidate self-promotion of application status/ai_score/phase via BEFORE UPDATE trigger

**Verify verdict:** yes  

**Rationale:** AUDIT P0-6: the policy `Candidates can update their own applications` (migrations/20251215031758_*.sql:2-6) is `FOR UPDATE USING(auth.uid()=candidate_id) WITH CHECK(auth.uid()=candidate_id)` with no column scoping (Postgres RLS cannot restrict columns), and the only BEFORE-UPDATE trigger on the table just stamps updated_at. So a candidate can PostgREST-UPDATE their own row and set ai_score/status/phase/ai_analysis/etc., defeating AI shortlisting and the autopilot phase gates.

Fix = a new BEFORE UPDATE trigger (SECURITY DEFINER) that only acts when the actor is the candidate (auth.uid()=NEW.candidate_id) AND is NOT the owning employer (jobs.employer_id=auth.uid()) AND NOT an active pipeline team member (reuses the existing public.is_team_member helper). Service-role / edge-function writes have auth.uid()=NULL, so they are never the candidate and pass through untouched — this is what preserves the real source of truth (trigger-ava-analysis writes ai_score/phase/status with the service-role client, index.ts:371-373).

For a candidate actor it RAISEs on any change (IS DISTINCT FROM, null-safe) to the pure verdict columns: ai_score, resume_score, ai_analysis, rejected_by, rejected_by_type. I verified across every candidate-facing page (Quiz/Sales/Chat*/Video/Voice/Typing/Portfolio/ApplicationForm/Wizard) that candidate code NEVER writes these (they write phase_ai_analysis — a different, legitimate self-report column — and they only SELECT ai_score). So hard-blocking these is zero functional cost.

status and phase are deliberately CONSTRAINED, not hard-blocked, because the live app has candidates legitimately writing them: ChatInterviewPhase.tsx:261 sets status:'reviewing'; ApplicationFormPhase.tsx:840 and CandidateApplicationWizard.tsx:894/904 set status:'pending'; auto-mode pages (ChatInterviewPhase:273, VideoIntroPhase:339, PortfolioUploadPhase:422, Wizard:893) advance phase. The trigger therefore allows a candidate status change only to 'pending' or 'reviewing' (plus no-op) — blocking the escalation values interview/offered/hired and self-forged rejected — and allows a candidate phase change only to a recognized phase: the sentinels application/quiz/review or an id that actually exists in that job's workflow_steps. -- =====================================================================
```sql
-- P0-7: Stop candidates forging employer signatures / integrity hashes
--       and tampering with locked, legally-binding signed documents.
--
-- The candidate UPDATE policy "Candidates can update document status"
-- (migration 20251214202144) had a USING clause but NO WITH CHECK, and
-- Postgres RLS cannot restrict *which columns* a candidate may write. So
-- a candidate could set status='signed', forge employer_signature_data /
-- employer_signed_at, overwrite *_hash integrity values, flip is_locked /
-- is_voided, and rewrite completion_certificate on their own document row.
--
-- Fix = (1) add a WITH CHECK so a candidate can only ever update rows that
-- belong to their own application (defence in depth; prevents re-homing a
-- row to another application via UPDATE), and (2) a BEFORE UPDATE trigger
-- that does the column-level enforcement RLS cannot: for *candidate*
-- updates it rejects any change to employer-signature / hash / lock /
-- void / certificate / final-signature columns, only permits status to
-- move to 'declined', and blocks every update once the document is locked.
--
-- The trigger is intentionally scoped to CANDIDATE updates only. Employer
-- and team-member updates (countersigning, locking, voiding, hashing,
-- editing) and service_role / edge-function updates are left untouched.
-- =====================================================================

-- ---------------------------------------------------------------------
-- 1. Recreate the candidate UPDATE policy WITH a WITH CHECK clause.
-- ---------------------------------------------------------------------

CREATE POLICY "Candidates can update document status"
ON public.documents
FOR UPDATE
USING (
  EXISTS (
    SELECT 1 FROM public.applications a
    WHERE a.id = documents.application_id
    AND a.candidate_id = auth.uid()
  )
)
WITH CHECK (
  EXISTS (
    SELECT 1 FROM public.applications a
    WHERE a.id = documents.application_id
    AND a.candidate_id = auth.uid()
  )
);

-- 2. BEFORE UPDATE trigger enforcing column-level immutability for
--    candidate-initiated updates.
-- ---------------------------------------------------------------------
CREATE OR REPLACE FUNCTION public.enforce_candidate_document_update_guard()
RETURNS TRIGGER
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_uid uuid := auth.uid();
  v_jwt_role text := COALESCE(
    NULLIF(current_setting('request.jwt.claims', true), '')::jsonb ->> 'role',
    ''
  );
  v_is_candidate boolean;
  v_is_employer_side boolean;
BEGIN
  -- Service role / internal (edge functions, autopilot, migrations,
  -- superuser): no end-user JWT context. Never restrict these.
  IF v_uid IS NULL OR v_jwt_role = 'service_role' THEN
    RETURN NEW;
  END IF;

  -- Is the caller the candidate on this document's application?
  SELECT EXISTS (
    SELECT 1 FROM public.applications a
    WHERE a.id = NEW.application_id
      AND a.candidate_id = v_uid
  ) INTO v_is_candidate;

  -- Is the caller the employer (or a permitted active team member) for
  -- this document's application? Employer-side updates are unrestricted.
  SELECT EXISTS (
    SELECT 1
    FROM public.applications a
    JOIN public.jobs j ON j.id = a.job_id
    WHERE a.id = NEW.application_id
      AND (
        j.employer_id = v_uid
        OR EXISTS (
          SELECT 1 FROM public.team_members tm
          WHERE tm.employer_id = j.employer_id
            AND tm.user_id = v_uid
            AND tm.status = 'active'
        )
      )
  ) INTO v_is_employer_side;

  -- Only candidates who are NOT also on the employer side are guarded.
  IF NOT v_is_candidate OR v_is_employer_side THEN
    RETURN NEW;
  END IF;

  -- ----- From here on, this is a pure candidate update. -----

  -- A locked (fully executed) document is immutable to candidates.
  IF COALESCE(OLD.is_locked, false) THEN
    RAISE EXCEPTION 'Document is locked and can no longer be modified'
      USING ERRCODE = 'check_violation';
  END IF;

  -- Reject any candidate change to employer-signature / integrity-hash /
  -- lock / void / certificate / final-signature / routing columns.
  IF NEW.employer_signature_data IS DISTINCT FROM OLD.employer_signature_data
     OR NEW.employer_signed_at   IS DISTINCT FROM OLD.employer_signed_at
     OR NEW.is_locked            IS DISTINCT FROM OLD.is_locked
     OR NEW.locked_at            IS DISTINCT FROM OLD.locked_at
     OR NEW.is_voided            IS DISTINCT FROM OLD.is_voided
     OR NEW.voided_at            IS DISTINCT FROM OLD.voided_at
     OR NEW.voided_reason        IS DISTINCT FROM OLD.voided_reason
     OR NEW.completion_certificate IS DISTINCT FROM OLD.completion_certificate
     OR NEW.signature_data       IS DISTINCT FROM OLD.signature_data
     OR NEW.signed_at            IS DISTINCT FROM OLD.signed_at
     OR NEW.document_hash        IS DISTINCT FROM OLD.document_hash
     OR NEW.final_pdf_hash       IS DISTINCT FROM OLD.final_pdf_hash
     OR NEW.v1_hash              IS DISTINCT FROM OLD.v1_hash
     OR NEW.v3_hash              IS DISTINCT FROM OLD.v3_hash
     OR NEW.sender_id            IS DISTINCT FROM OLD.sender_id
     OR NEW.recipient_id         IS DISTINCT FROM OLD.recipient_id
     OR NEW.application_id        IS DISTINCT FROM OLD.application_id
     OR NEW.document_code        IS DISTINCT FROM OLD.document_code
  THEN
    RAISE EXCEPTION 'Candidates cannot modify employer signature, integrity, lock, void, or routing fields on a document'
      USING ERRCODE = 'insufficient_privilege';
  END IF;

  -- Candidates may move status only into 'declined' (the decline flow).
  -- They may never set 'signed' (that is the employer countersignature).
  IF NEW.status IS DISTINCT FROM OLD.status AND NEW.status <> 'declined' THEN
    RAISE EXCEPTION 'Candidates may only decline; the signed status is set on employer countersignature'
      USING ERRCODE = 'insufficient_privilege';
  END IF;

  RETURN NEW;
END;
$$;

CREATE TRIGGER enforce_candidate_document_update_guard
BEFORE UPDATE ON public.documents
FOR EACH ROW
EXECUTE FUNCTION public.enforce_candidate_document_update_guard();
```

```sql
      USING ERRCODE = '22023';
  END IF;

  -- we read straight from auth.users.
  SELECT lower(u.email) INTO v_caller_email
  FROM auth.users u
  WHERE u.id = v_uid
    AND u.email_confirmed_at IS NOT NULL;

  IF v_caller_email IS NULL THEN
    RAISE EXCEPTION 'A verified email is required to join a team'
      USING ERRCODE = '28000';
  END IF;

  -- Lock the invitation row to make the lookup + accept atomic and prevent a
  -- concurrent double-accept.
  SELECT * INTO v_inv
  FROM public.team_invitations ti
  WHERE ti.invite_code = p_code
  FOR UPDATE;

  IF NOT FOUND THEN
    RAISE EXCEPTION 'Invitation not found'
      USING ERRCODE = 'P0002';
  END IF;

  -- The invitation must still be open.
  IF v_inv.status <> 'pending'::public.invitation_status THEN
    RAISE EXCEPTION 'This invitation has already been used'
      USING ERRCODE = 'P0001';
  END IF;

  IF v_inv.expires_at < now() THEN
    RAISE EXCEPTION 'This invitation has expired'
      USING ERRCODE = 'P0001';
  END IF;

  -- The caller's verified email MUST match the invited email. invitee_email is
  -- NOT NULL on team_invitations, so there is no "unrestricted" invite path.
  IF lower(v_inv.invitee_email) <> v_caller_email THEN
    RAISE EXCEPTION 'This invitation is for a different email address'
      USING ERRCODE = '42501';
  END IF;

  -- Insert the membership. Every privileged column is taken from the
  -- invitation row, never from client input. employer_id = the inviter.
  -- If the user is already a member of this employer, surface a friendly error.
  BEGIN
    INSERT INTO public.team_members (
      user_id,
      employer_id,
      invitation_id,
      name,
      email,
      department,
      permission_level,
      can_create_jobs,
      can_delete_jobs,
      can_message_candidates,
      can_manage_pipeline,
      can_schedule_interviews,
      can_send_documents,
      assigned_job_ids,
      status,
      onboarding_completed
    )
    VALUES (
      v_uid,
      v_inv.inviter_id,
      v_inv.id,
      COALESCE(v_inv.invitee_name, ''),
      v_inv.invitee_email,
      v_inv.department,
      v_inv.permission_level,
      v_inv.can_create_jobs,
      v_inv.can_delete_jobs,
      v_inv.can_message_candidates,
      v_inv.can_manage_pipeline,
      v_inv.can_schedule_interviews,
      v_inv.can_send_documents,
      COALESCE(v_inv.assigned_job_ids, '{}'::uuid[]),
      'active',
      false
    );
  EXCEPTION
    WHEN unique_violation THEN
      RAISE EXCEPTION 'You are already a member of this team'
        USING ERRCODE = '23505';
  END;

  -- Mark the invitation accepted (same transaction).
  UPDATE public.team_invitations
  SET status = 'accepted'::public.invitation_status
  WHERE id = v_inv.id;

  -- Ensure the team_member role exists for this user (idempotent).
  INSERT INTO public.user_roles (user_id, role)
  VALUES (v_uid, 'team_member'::public.app_role)
  ON CONFLICT (user_id, role) DO NOTHING;

  RETURN jsonb_build_object(
    'success', true,
    'employer_id', v_inv.inviter_id,
    'invitation_id', v_inv.id
  );
END;
```

```sql
-- Buckets `resumes`, `videos`, `portfolios`, `message-attachments` were
-- created with public=true. Public buckets serve via an unauthenticated CDN
-- path (/storage/v1/object/public/...) that BYPASSES storage.objects RLS,
-- so the scoped SELECT policies on those buckets were dead code and anyone
-- with (or guessing) an object URL could download resumes, video intros,

--   1. Flips the four PII buckets to public=false (avatars stays public -
--      avatars are intended to be world-readable).
--   2. Drops the "publicly readable" SELECT policies.
--   3. Re-scopes message-attachments INSERT/SELECT/DELETE to the owner
--      (and, for SELECT, the message recipient) instead of bucket-only.
--   4. Adds owner + reviewer (employer / active team member) SELECT
--      policies on resumes and portfolios so employer review keeps working
--      via short-lived signed URLs (videos already has an employer policy;
--      a team-member policy is added for parity).
--

-- /object/public/ links stop resolving. App call sites that must switch
-- from getPublicUrl() to createSignedUrl() are listed in the fix notes.
-- The server-side resume fetch (_shared/resume.ts) and ava-voice-session
-- already fall back to admin createSignedUrl(), so the AI analysis path is
-- unaffected.
-- =====================================================================

-- 1. Make the PII buckets private (idempotent).
-- ---------------------------------------------------------------------
UPDATE storage.buckets
SET public = false
WHERE id IN ('resumes', 'videos', 'portfolios', 'message-attachments');

-- 2. Drop the world-readable / bucket-only SELECT policies.
--    (DROP ... IF EXISTS is idempotent and safe to re-run.)
-- ---------------------------------------------------------------------

-- resumes: "Resumes are publicly readable" (no TO, no owner check) -> P0-9
DROP POLICY IF EXISTS "Resumes are publicly readable" ON storage.objects;

-- portfolios: "Portfolio files are publicly accessible" (no owner check) -> P0-9
DROP POLICY IF EXISTS "Portfolio files are publicly accessible" ON storage.objects;

-- message-attachments: all three policies were scoped only by bucket_id
-- (no auth.uid()/folder check) -> P1-1: cross-tenant read/write/DELETE.
DROP POLICY IF EXISTS "Users can upload message attachments" ON storage.objects;
DROP POLICY IF EXISTS "Users can view message attachments" ON storage.objects;
DROP POLICY IF EXISTS "Users can delete their message attachments" ON storage.objects;

-- 3. message-attachments: owner-scoped INSERT/DELETE + participant SELECT.
--    Path layout is `${auth.uid()}/<file>` (useMessages.ts:346), so the
--    first folder segment is the uploader. SELECT also allows the other
--    party of a message that references the object (sender or receiver).
-- ---------------------------------------------------------------------

CREATE POLICY "Users can upload their own message attachments"
ON storage.objects FOR INSERT TO authenticated
WITH CHECK (
  bucket_id = 'message-attachments'
  AND auth.uid()::text = (storage.foldername(name))[1]
);

CREATE POLICY "Users can delete their own message attachments"
ON storage.objects FOR DELETE TO authenticated
USING (
  bucket_id = 'message-attachments'
  AND auth.uid()::text = (storage.foldername(name))[1]
);

-- Owner can read their own uploads.
CREATE POLICY "Users can view their own message attachments"
ON storage.objects FOR SELECT TO authenticated
USING (
  bucket_id = 'message-attachments'
  AND auth.uid()::text = (storage.foldername(name))[1]
);
```

```sql
import { supabase } from "@/integrations/supabase/client";

/**
 * Parse a stored Supabase Storage reference into { bucket, path }.
 * Accepts either a full storage URL (public or signed) or a bare object path.
 * Mirrors supabase/functions/_shared/resume.ts::parseStorageUrl so the same
 * stored values resolve identically on client and server.
 */
export function parseStorageRef(
  bucket: string,
  urlOrPath: string,
): { bucket: string; path: string } | null {
  if (!urlOrPath) return null;

  // Full storage URL: /storage/v1/object/(public|sign)/<bucket>/<path>
  const match = urlOrPath.match(
    /\/storage\/v1\/object\/(?:public\/|sign\/)?([^/]+)\/([^?]+)/,
  );
  if (match) {
    return { bucket: match[1], path: decodeURIComponent(match[2]) };
  }

  // Bare path, optionally prefixed with the bucket id.
  const stripped = urlOrPath.startsWith(`${bucket}/`)
    ? urlOrPath.slice(bucket.length + 1)
    : urlOrPath;
  return { bucket, path: stripped };
}

/**
 * Mint a short-lived signed URL for a private-bucket object that was stored
 * either as a full URL (legacy getPublicUrl value) or as a bare path.
 * Returns null if the reference can't be resolved or signing fails.
 *
 * The caller must satisfy a storage.objects SELECT RLS policy for the object
 * (owner, employer-of-job, active assigned team member, or message
 * participant) - signing does not bypass RLS.
 */
export async function getSignedStorageUrl(
  bucket: string,
  urlOrPath: string | null | undefined,
  expiresInSeconds = 3600,
): Promise<string | null> {
  if (!urlOrPath) return null;

  const ref = parseStorageRef(bucket, urlOrPath);
  if (!ref) return null;

  const { data, error } = await supabase.storage
    .from(ref.bucket)
    .createSignedUrl(ref.path, expiresInSeconds);

  if (error || !data?.signedUrl) {
    console.error(`[signedUrl] failed to sign ${ref.bucket}/${ref.path}`, error);
    return null;
  }
  return data.signedUrl;
}
```

```
        } else {
          return new Response(
            JSON.stringify({ error: 'Unknown or expired voice session' }),
            { status: 404, headers: { ...corsHeaders, 'Content-Type': 'application/json' } }
          );
        }
      } else {
        voiceSessionRow = session;

        // The session's billed owner must match the target resolved from auth/applicationId.
        if (session.voice_owner_user_id !== targetUserId) {
          console.error(`[deduct-voice-minutes] Owner mismatch: session owner ${session.voice_owner_user_id} != target ${targetUserId}`);
          return new Response(
            JSON.stringify({ error: 'Session does not belong to this account' }),
            { status: 403, headers: { ...corsHeaders, 'Content-Type': 'application/json' } }
          );
        }

        // Already finalized -> replay/double-bill attempt. Refuse and report 0 deducted.
        if (session.finalized_at) {
          console.warn(`[deduct-voice-minutes] Session ${sessionId} already finalized at ${session.finalized_at}; refusing re-deduction`);
          return new Response(
            JSON.stringify({ success: true, minutesDeducted: 0, alreadyFinalized: true, targetUserId, mode }),
            { status: 200, headers: { ...corsHeaders, 'Content-Type': 'application/json' } }
          );
        }
      }
    }

    if (voiceSessionRow) {
      // Authoritative duration = server elapsed since token mint, ceil to the minute.
      const elapsedMs = Date.now() - new Date(voiceSessionRow.server_started_at).getTime();
      let serverMinutes = Math.max(1, Math.ceil(elapsedMs / 60000));
      // Hard cap per session (defends against a tab left open before disconnect).
      if (voiceSessionRow.max_billable_minutes && voiceSessionRow.max_billable_minutes > 0) {
        serverMinutes = Math.min(serverMinutes, voiceSessionRow.max_billable_minutes);
      }
      billableMinutes = serverMinutes;

      // Atomically claim the session so concurrent calls cannot both bill it.
      const { data: claimed, error: claimError } = await supabaseAdmin
        .from('voice_sessions')
        .update({
          finalized_at: new Date().toISOString(),
          billed_minutes: billableMinutes,
          client_reported_minutes: clientReportedMinutes,
        })
        .eq('id', voiceSessionRow.id)
        .is('finalized_at', null)
        .select('id');

      if (claimError || !claimed || claimed.length === 0) {
        console.warn(`[deduct-voice-minutes] Session ${sessionId} was concurrently finalized; refusing re-deduction`);
        return new Response(
          JSON.stringify({ success: true, minutesDeducted: 0, alreadyFinalized: true, targetUserId, mode }),
          { status: 200, headers: { ...corsHeaders, 'Content-Type': 'application/json' } }
        );
      }

      if (clientReportedMinutes !== null && clientReportedMinutes < billableMinutes) {
        console.warn(`[deduct-voice-minutes] Client under-reported (${clientReportedMinutes}m) vs server (${billableMinutes}m); billing server value`);
      }
    } else {
      // Legacy fallback path (no anchor): trust client minutes but still cap defensively.
      billableMinutes = Math.min(clientReportedMinutes as number, 120);
      console.warn(`[deduct-voice-minutes] legacy_no_session path: billing client-reported ${billableMinutes}m`);
    }

    console.log(`[deduct-voice-minutes] Mode: ${mode}, Deducting ${billableMinutes} minutes from user ${targetUserId}`);
```
